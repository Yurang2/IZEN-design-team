# 01_마에스트로 (대화 정리 문서)

## 0) 메타 요약 (강약조절)
- `강 (지금 결정)`:
  - Tier1은 시각 제작 자동화보다 `문서/카피/브리프/다국어` 자동화가 ROI가 가장 크다.
  - 새 탭은 즉시 추가하지 않고, 기존 탭 내 `AI 작성` 액션(모달/드로어)으로 시작한다.
  - GPT API 키는 Worker Secret로만 관리하고, AI 출력은 반드시 Draft + 사람 승인으로 운영한다.
- `중 (이번 분기 권고)`:
  - Notion DB를 `AI_Requests`, `AI_Drafts`, `AI_Templates` 중심으로 먼저 신설한다.
  - docs/hook-master/ops 문서를 먼저 갱신하고 API 계약 후 구현한다.
  - 정보 과다 완화를 위해 1차 화면에는 핵심 CTA만 남기고 운영/디버그 정보는 접거나 분리한다.
- `약 (관찰/후순위)`:
  - 전용 AI 탭 승격은 사용량 KPI 충족 후 판단한다.
  - IFU/규제 문구 자동화는 인허가 승인 체계와 묶어서 단계적으로 확장한다.

## 1) 문서 목적
- 이 문서는 사용자와의 대화에서 도출된 `현황 진단`, `AI 도입 방향`, `실행 우선순위`를 한 번에 확인하기 위한 정리본이다.
- 본 문서는 코드/문서 수정 지시가 아닌, 전략 및 설계 판단을 위한 기준 메모다.

## 2) 대화 배경
- 사용자 요청:
  - 현재 배포 코드가 Asana/Monday급 SaaS 대비 부족한 점 리뷰
  - 특히 `정보 과다` 문제와 docs 지침 보완점 점검
  - 실제 수정은 하지 않고 리뷰만 진행
- 이후 추가 요청:
  - 조직도/업무 범위를 기준으로 AI가 가장 큰 임팩트를 내는 파이프라인 분석
  - 디자인팀 기준으로 대체/자동화 난이도 순서 제안
  - Notion DB 신설 + 웹에서 즉시 작성 + GPT API 연동 시 탭 전략 및 지침 적합성 검토

## 3) 코드/문서 리뷰 핵심 결과

### 3-1) 제품 상태 진단 (요약)
- 강점:
  - Notion 원장(SSOT) 기반 데이터 운영 방향이 명확함
  - Worker 중심 보안/캐시/연동 구조가 정리되어 있음
  - 프로젝트/업무/체크리스트/회의록 기능이 이미 확장되어 있음
- 핵심 문제:
  - 화면/기능/운영 정보가 한 번에 과다 노출되어 인지 부하가 큼
  - 문서와 코드 사이 일부 정책 충돌(예: 인증 게이트) 및 문서 과밀/혼재

### 3-2) 정보 과다의 주요 원인
- 상단 툴바에 사용자 작업 + 운영/디버그 기능이 동시에 노출됨
  - 예: 새 업무, 새로고침, API 테스트, 수동 Export, Theme, API Base, 마지막 동기화
- `App.tsx` 단일 파일에 라우팅/권한/가이드/뷰 조립이 과집중됨
- 프로젝트/체크리스트/회의록 화면에서 카드/필터/지표/액션이 동시 노출됨
- 가이드/런북/이관 로그가 README에 과다 집적되어 목적별 문서 분리가 약함

### 3-3) docs 지침 보완 필요점
- `정보 예산` 규칙 부재:
  - 1차 화면 노출 요소 수, 고급 정보 접기 규칙, 실무자/관리자 표시 분리 기준 필요
- 테스트/수용 문서 실체화 필요:
  - `ops/30_TEST_PLAYBOOK.md`, `ops/40_ACCEPTANCE_CHECKLIST.md`를 실제 시나리오로 채워야 함
- README 분리 필요:
  - Quickstart / Runbook / Changelog 분리로 독자별 맥락 분리

## 4) 조직/업무 기준 AI 적용 인사이트

### 4-1) 전사 파이프라인에서 AI 고임팩트 구간
1. 문서 변환 구간:
   - 요청/회의 -> 보고/품의/제안/증빙
2. 다국어 변환 구간:
   - 해외영업 요구 -> 시장별 카피/자료 버전
3. 부서 간 핸드오프 구간:
   - 서울(해외영업/마케팅) <-> 화성(연구/품질/생산) 조율 문서 정리

### 4-2) 디자인팀 기준 병목 해소 포인트
- 병목 본질:
  - 시각 제작 자체보다 `전후 텍스트/요청정리/다국어/행정` 작업이 병목
- AI 우선 담당 영역:
  1. 품의/지출결의/보고/제안서 초안
  2. Certificate 변수 치환형 문서
  3. PPT/뉴스레터/팝업 카피 및 구조안
  4. 요청 브리프 정규화(요청 원문 -> 구조화)
  5. 다국어 현지화(용어집/금칙어 기반)

## 5) 작업 난이도 분류 (쉬운 순 -> 어려운 순)

### Tier 1 (즉시 적용 가능)
- 품의서/지출결의서/보고서/제안서 등 정형 문서 초안
- Certificate 템플릿 치환
- PPT 스토리라인/슬라이드 텍스트 초안
- 뉴스레터/팝업 카피 초안

### Tier 2 (빠른 도입 가능)
- IFU 초안 보조(인허가 검토 전제)
- 포스터/리플렛 텍스트 및 구성안
- SNS 기획/카피/캘린더, 홈페이지 업데이트 문안
- 미팅 사전 안건/사후 회의록 구조화

### Tier 3 (부분 대체)
- 브로슈어/카탈로그 콘텐츠 아키텍처
- 배너/현수막/부스 그래픽 카피 구조화
- 패키지 규제 표기 텍스트 정리
- 이벤트/판촉물 운영 문서

### Tier 4 (대체 어려움, 보조 중심)
- 2D/3D 모션 제작, 3D 렌더링, 영상 편집, 촬영 실무

## 6) Tier1 구현 전략 (현재 합의안)

### 6-1) 기능 우선순위
1. `AI Brief Copilot`
   - 요청 원문 -> 표준 브리프 + 누락 질의 자동 생성
2. `AI Copy Copilot`
   - 채널별 카피 초안 생성(포스터/팝업/SNS/PPT)
3. `AI Localization Copilot`
   - 언어별 번역 + 용어집 준수 + 리스크 표현 경고
4. `AI Doc Copilot`
   - 품의/지출/결과보고 문서 초안 생성

### 6-2) 운영 원칙
- AI 출력은 항상 Draft 상태로 저장
- 최종 승인자는 사람(디자인팀/마케팅/인허가)
- 프롬프트/출력/수정 이력 로깅
- GPT API 키는 Worker Secret로만 보관(프론트 노출 금지)

### 6-3) 도입 순서 (강/중/약)
- `강 (즉시 착수)`:
  1. AI Brief Copilot
  2. AI Doc Copilot
- `중 (2차 착수)`:
  1. AI Copy Copilot
  2. AI Localization Copilot
- `약 (확장 단계)`:
  1. IFU 고도화
  2. 전용 AI 탭 분리

## 7) Notion DB 신설 및 UI 연동 전략

### 7-1) DB 최소안
- `AI_Requests`:
  - 요청 원문, 요청팀, 산출물 타입, 언어, 납기, 상태, 관련 업무(Relation)
- `AI_Drafts`:
  - request relation, 버전, 모델, 출력문, 검토자, 승인상태
- `AI_Templates`:
  - 시스템 프롬프트, 출력 형식, 사용 범위
- `AI_Glossary` (선택):
  - 의료/브랜드 용어, 금칙어, 권장 표현

### 7-2) 탭 신설 여부 (권고)
- 1단계:
  - `새 탭 즉시 추가 X`
  - 기존 탭(업무/회의록/가이드) 내 `AI 작성` 버튼 + 모달/드로어로 시작
- 2단계(승격 조건 충족 시):
  - 요청량, 승인 흐름, 버전 관리가 독립 워크플로로 고정될 때 전용 탭 신설
- 이유:
  - 현재 UI가 이미 과밀하며, 상위 탭 추가는 정보 과다를 악화시킬 가능성이 큼
