# IZEN 운영툴 설계도 (SSOT)

## 1) 문서 목적
- 이 문서는 `Yurang2/IZEN-design-team`의 단일 설계 기준(SSOT)이다.
- 목표는 "Notion 기반 운영 데이터"를 웹에서 안전하게 조회/수정하고, 팀 의사결정 속도를 높이는 것이다.
- 코드보다 계약(문서/타입/API)을 우선하며, 기능 추가 시 이 문서를 먼저 갱신한다.

## 2) 제품 한 줄 정의
- 본 제품은 `행사/전시/프로젝트 운영`을 위한 내부 운영 콘솔이며, 프로젝트-업무-체크리스트-회의록을 하나의 흐름으로 연결한다.

## 2-1) 최종 목표 (비즈니스 관점)
- 최종 목표는 아래 3개 관점을 각각 최고 수준으로 만족시키는 뷰를 만드는 것이다.
1. 보고용 뷰: 경영/공유/의사결정 보고에 최적화된 요약과 근거를 제공한다.
2. 운영용(팀장용) 뷰: 일정/리스크/우선순위/리소스 조정 판단에 최적화한다.
3. 업무용 뷰: 실무자의 실행/수정/완료 처리를 가장 빠르고 정확하게 지원한다.

## 3) 핵심 원칙
- 원칙 1: 사람이 편집하는 운영 데이터의 정답은 Notion DB에 둔다. 앱 상태/대용량 바이너리는 D1/R2에 둔다.
- 원칙 2: 보안/세션 정책은 유지한다. (`docs/WORKER_SECURITY.md`)
- 원칙 3: 장애 시 복구 절차를 우선한다. (`docs/RECOVERY_CLOUDFLARE_HOSTING.md`)
- 원칙 4: 기능 변경은 MD(설계/계약) 먼저, 코드 나중.
- 원칙 5: 1개 기능 단위로 경계를 쪼개고, 덧붙이기보다 재작성 원칙을 우선한다.

## 3-1) 노션 데이터 정책
- Notion은 원장(SSOT) 데이터 저장소이며, 스토리보드 문서 상태는 D1/R2를 원장으로 쓴다.
- 웹은 Notion의 Raw 데이터를 직접 노출하지 않는다.
- Worker는 Notion 데이터를 내부 계약(ViewModel)로 정규화(normalize)한다.
- 웹은 이 정규화된 ViewModel을 기준으로 동작한다.
- Notion 컬럼명은 내부 타입 계약(`worker/src/types.ts`)을 통해 매핑된다.

## 4) 범위 정의 (In / Out)
### In Scope
- 프로젝트/업무 조회 및 수정
- 행사 체크리스트 계산/할당/상태 관리
- 회의록 업로드/전사/화자 매핑
- 운영 문서(사용법/보안/복구) 제공

### Out of Scope (현재)
- 외부 고객용 퍼블릭 서비스
- ERP/회계 등 타 시스템 완전 통합
- 복잡한 권한 모델(RBAC 세분화) [TBD]

## 5) SSOT 데이터 원장
- Project 원장: Notion Project DB (`NOTION_PROJECT_DB_ID`)
- Task 원장: Notion Task DB (`NOTION_TASK_DB_ID`)
- Checklist 원장: Notion Checklist DB (`NOTION_CHECKLIST_DB_ID`)
- Checklist 할당 원장: Notion 할당 매트릭스 DB (`NOTION_CHECKLIST_ASSIGNMENT_DB_ID`, 옵션)
- Meetings 원장: Notion Meeting DB (`3f3c1cc7ec278216b5e881744612ed6b`, 고정)
- Reference 원장: Notion Reference DB (`NOTION_REFERENCE_DB_ID`, 옵션)
- Storyboard 원장: Notion 메타데이터 + D1 본문 + R2 이미지
- 음성 원본 저장소: Cloudflare R2 (`MEETING_AUDIO_BUCKET`)

## 6) 도메인 경계 (MECE)
### A. 프로젝트/업무 도메인
- 프로젝트 목록/메타 정보 조회
- 업무 CRUD 및 상태 업데이트
- 타입 계약 기준: `worker/src/types.ts`의 `ProjectRecord`, `TaskRecord`, `CreateTaskInput`, `UpdateTaskInput`

### B. 체크리스트 도메인
- 행사 기준 체크리스트 계산(영업일/기준일 반영)
- 할당 상태(`assigned | unassigned | not_applicable`) 관리
- 타입 계약 기준: `ChecklistPreviewItem`, `ChecklistAssignmentRow`, `ChecklistAssignmentStatus`

### C. 회의록 도메인
- 오디오 업로드(R2 presigned URL) -> AssemblyAI 전사 -> webhook 완료 반영
- 화자 라벨(A/B/C) 보존 + 표시명 매핑
- 회의록 본문은 Notion 페이지 블록으로 저장
- 키워드 부스트 정책은 운영 결정에 따라 On/Off 가능 [TBD]

### D. 공통 인프라 도메인
- 인증/세션/레이트리밋/CORS/보안 헤더
- 캐시/복구/배포 운영

### E. 행사 그래픽 운영 도메인
- 호텔/세미나형 행사는 `시간 기준 타임테이블`로 운영한다.
- 전시회/부스형 행사는 `상황 기준 운영표(playbook)`로 운영한다.
- 동일한 내부 화면에서 `자체행사 / 전시회` 모드를 전환하되, 외부 공유는 시간표 중심으로 유지한다.

## 7) 사용자 흐름 (핵심)
1. 운영자가 로그인한다.
2. 프로젝트/업무 현황을 확인하고 업무 상태를 갱신한다.
3. 행사 체크리스트 탭에서 일정/할당/위험 신호를 점검한다.
4. 회의록 탭에서 음성 업로드 후 화자 매핑을 저장한다.
5. 필요 시 내보내기/보고 자료를 생성한다.

## 8) 아키텍처 개요
- Frontend: React + Vite (Cloudflare Pages)
- Backend: Cloudflare Workers (`/api/*`)
- Data: Notion API + (일부 보조 저장/캐시) + R2
- 운영 계약: Env/타입/API 계약을 기준으로 동작

## 9) 보안/운영 기준 (고정)
- 인증/세션/보안 헤더/CORS/레이트리밋: `docs/WORKER_SECURITY.md` 준수
- 복구/배포 기준: `docs/RECOVERY_CLOUDFLARE_HOSTING.md` 준수
- 기본 정책: 민감 경로 `Cache-Control: no-store`, 비밀값은 Secret 저장

## 10) API 계약의 기준점
- API 계약의 1차 기준은 Worker 구현 및 타입 정의다.
- 기준 파일:
  - `worker/src/types.ts`
  - `worker/src/index.ts`
- 문서-코드 불일치 시, 이 설계도와 hook-master를 먼저 수정하고 코드를 맞춘다.

## 11) 성공 기준 (Definition of Done)
- 기능 단위 변경마다 아래를 만족해야 한다.
1. 설계 문서(본 문서 + hook 문서) 반영 완료
2. API 계약/타입 계약 불일치 없음
3. `npm run build` 성공
4. 핵심 흐름 수동 테스트 완료 (조회/수정/에러 처리)
5. 보안 정책 회귀 없음

## 12) 현재 결정 사항
- 데이터 원장은 Notion 중심으로 유지한다.
- 회의록 오디오는 R2를 사용하며 별도 서버 프로세스는 두지 않는다.
- 체크리스트 할당 상태는 "행사-체크리스트 할당 매트릭스" 기준으로 관리한다.

## 2026-02-27 Upload Runtime Update
- Current meeting upload pipeline: Browser -> (R2 presigned or worker_direct fallback) -> R2 -> AssemblyAI(audio_url) -> webhook -> transcript detail/publish.
- Upload timeout handling was updated: dynamic timeout by file size (min 5m, max 30m).
- Upload retry policy was updated: retry once on retryable upload errors (total up to 2 attempts).
- worker_direct is a fallback path when R2 presigned URL is not available. Hard size blocking was removed; warning-only behavior remains.
- Deployment note: manual Cloudflare Pages deploy via Wrangler requires CLOUDFLARE_API_TOKEN in non-interactive environments.
- 이미지 첨부 UI는 파일 선택과 드래그앤드롭 업로드를 기본 지원한다.
