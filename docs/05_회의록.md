# 05_회의록 페이지 지침

## 1) 목적
- 회의록 탭은 `전사 결과 확인 -> 화자 라벨링 -> Notion 반영`의 3단계를 분리한다.
- 라벨링 전 결과(A/B/C)는 내부 검토용이며, Notion 최종 본문에는 라벨 확정 결과만 기록한다.

## 2) 데이터 저장 원칙
- 별도 중간 DB를 두지 않는다.
- 중간 상태는 Notion transcript row의 속성으로 관리한다.
- 필수 속성:
- `assemblyId`, `status`, `speakerMapJson`, `bodySynced`, `errorMessage`, `textPreview`

## 3) 상태 표시 원칙 (사용자 혼란 방지)
- 화면 표시 상태는 아래처럼 분리한다.
- `전사 진행중`: queued/submitted/processing
- `라벨링 필요`: AssemblyAI 전사 completed 이지만 미매핑 화자 존재 또는 Notion 미반영
- `반영 완료`: 화자 매핑 완료 + Notion 반영 완료(`bodySynced=true`)
- 주의: 단순 전사 완료(completed)를 곧바로 최종 완료로 표기하지 않는다.

## 4) 화자 매핑/반영 규칙
- `매핑 저장`:
- A/B/C -> 실명 매핑을 `speakerMapJson`에 저장만 한다.
- Notion 본문은 이 단계에서 쓰지 않는다.
- `Notion 반영`:
- 매핑 누락 화자가 없어야 실행 가능하다.
- 실행 시 Notion 페이지 본문을 재작성한다.
- `전문`에는 `화자별 발화`만 기록하고, `원문 텍스트` 섹션은 생성하지 않는다.
- `화자별 발화`는 `[HH:MM:SS-HH:MM:SS] 화자: 발화` 형식 타임스탬프를 포함한다. (시각 정보가 없으면 생략)
- 같은 `transcriptId`에 대해 `매핑 저장 + Notion 반영`은 반복 실행 가능하다.
- 이 반복 실행은 AssemblyAI 재전사를 만들지 않으므로 전사 비용을 추가로 발생시키지 않는다.

## 5) 제목 파싱 규칙
- 업로드 단계에서 별도 `회의 제목` 입력은 받지 않고, 파일명만 사용한다.
- 파일명(또는 내부 title)이 `yymmdd <제목>` 형식이면:
- Notion `날짜` = `20yy-mm-dd`
- Notion 페이지 `이름(제목)` = `<제목>` 부분만 사용
- 형식 미일치 시 전체 문자열을 제목으로 사용하고 날짜는 비워둔다.
- R2 key에 UUID prefix(`<32hex>-`)가 붙은 경우 prefix 제거 후 파싱한다.
- Notion 날짜 컬럼은 `날짜` 또는 `일자`를 자동 인식해 쓴다.

## 6) 오디오 첨부 규칙
- Notion 페이지 최상단에 원본 녹음 접근 경로를 둔다.
- 구현 옵션:
- A안(권장): R2 파일을 Notion 파일로 업로드해 첨부(영구 접근성 유리)
- B안: R2 공개 URL 또는 외부 링크 블록 첨부(서명 URL 만료/권한 이슈 점검 필요)
- 보안상 비공개 버킷이면 만료 없는 직접 URL 첨부는 지양한다.

## 7) 최종 반영 조건
- 아래 조건을 모두 충족해야 `반영 완료`로 간주한다.
- 전사 상태 `completed`
- 모든 화자 라벨 매핑 완료
- Notion 본문 반영 성공 (`bodySynced=true`)
- `OPENAI_API_KEY` 설정 시 `요약` 섹션 자동 생성 성공(실패 시 placeholder 또는 빈 요약 허용 정책 적용)

## 8) GPT Draft Prompt Policy (2026-02-27)
- Summary generation uses `gpt-5-mini` by default.
- GPT input source is timestamped utterance lines:
  - `[HH:MM:SS-HH:MM:SS] Speaker: utterance`
- Draft output policy enforces:
  - explicit timestamp evidence for decisions/requests/changes/deadlines/risks/role assignment
  - `[Uncertain]` marker when evidence is ambiguous
  - no unsupported claims outside source utterances
- The summary section is treated as a review draft, not a final confirmed record.
- Prompt enforces output language as Korean (`ko-KR`).
- Publish response returns summary status diagnostics: `summaryGenerated`, `summaryError`.
- If summary generation fails, Notion `요약` 섹션 now records a failure hint instead of only generic placeholder.
- Summary generation auto-retries once on token-incomplete responses (`incomplete=max_output_tokens`).
- Retry uses shorter source and larger output-token budget to reduce empty-summary failure.
- Summary post-processing now normalizes common English headings/labels to Korean.
- If first response is partial (`incomplete=max_output_tokens`) even with text, Worker retries and prefers retry output.
- GPT summary output contract is now fixed to GFM headings and ordered sections.
- Server validates required markdown headers; if invalid, it retries once and auto-patches missing headers.
- Notion summary rendering maps markdown prefixes to blocks: `##`=>heading_2, `###`=>heading_3, `-`=>bulleted list.
- Replace `메타` heading with `회의 개요` to avoid naming conflict.
- Summary compactness rule: participants as names only, short draft-note, table-only for `정해진 내용 / 확인 필요`, and no duplicate prose.
- No item-count limit for `핵심 안건 요약` and `불확실/추가 확인 필요 구간`.
- Speaker mapping now rejects single-letter alphabet display names (`A`~`Z`, case-insensitive).
- Publish path also blocks legacy mappings containing single-letter alphabet names.

## 9) 2026-02-27 Summary/Transcript Rendering Update
- `## 참여자별 해야 할 일` 섹션은 실제 담당자 액션이 있을 때만 출력한다.
- `## 불확실/추가 확인 필요 구간` 섹션은 실제 불확실 항목이 있을 때만 출력한다.
- 위 두 섹션은 내용이 없으면 생성하지 않는다. 더미 문구(예: 자동 보정 안내)는 금지한다.
- Notion `전문` 섹션에서 `status=... generated_at=...` 메타 라인은 출력하지 않는다.
- 오디오 파일은 페이지 본문 최상단 파일 블록 + DB 컬럼(`Audio File`)에 함께 첨부한다.

## 10) 업로드 단계 가시화/타임아웃 (2026-02-27)
- 업로드 플로우는 UI에서 3단계로 표시한다.
  - `1/3 업로드 준비` (`POST /api/uploads/presign`)
  - `2/3 파일 업로드` (`PUT presign.putUrl`)
  - `3/3 전사 요청 생성` (`POST /api/transcripts`)
- 각 단계별 타임아웃:
  - presign: 20초
  - 파일 업로드: 5분
  - 전사 요청 생성: 45초
- 업로드 진행 중 `요청 취소` 버튼으로 클라이언트 요청을 중단할 수 있다.
- 단계 타임아웃/취소 시 사용자에게 단계별 안내 문구를 명시한다.
